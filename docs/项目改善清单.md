- [ ] **邮件 SMTP 密码** — `password: yvustyqjhgpbehae` 明文

**改进方式**：使用 `spring-cloud-config` / `jasypt` 加密，或通过环境变量 `SPRING_DATASOURCE_PASSWORD` 注入。

### 注解冲突

## P2 — 性能 · 建议迭代优化

### N+1 缓存查询

- [ ] `pageArticles()` 中先查缓存列表，再逐个查单条缓存。如果本地缓存未命中叠加 Redis 未命中，极端情况 N=30 时产生大量请求
- [ ] 考虑批量查询（mget） + 本地缓存预热

### 深分页

- [ ] MySQL 分页随着页码增大性能下降，`limit 100000, 10` 类型查询
- [ ] 改为游标分页（cursor-based）或使用 ES 做分页查询

### 连接池 & 超时配置

- [ ] 没有显式配置 HikariCP 连接池参数（最大连接数、超时等），生产环境需要用默认值
- [ ] Redis 连接池参数可优化

### 静态资源

- [ ] 图片目前走 MinIO 直连，没有 CDN 或反向代理缓存，大流量场景会成为瓶颈

## P3 — 架构/设计 · 建议中期重构

### GlobalExceptionHandler 位置不当

- [ ] 异常处理器在 `myblog-admin` 模块，但其他模块也需要异常处理
- [ ] 应该移到 `myblog-common` 并用 `@ComponentScan` 共享

### 响应类型混用

- [ ] 部分 Controller 返回 `Response<T>`，部分返回 `ResponseEntity<Response<T>>`（如 `myblog-common` 中的 GlobalExceptionHandler）
- [ ] 统一为一种风格，降低前端解析复杂度

### 无 API 版本控制

- [ ] 所有接口无版本前缀（`/api/v1/`），后续兼容升级困难
- [ ] 建议加 `@RequestMapping("/api/v1/...")`

### DTO / VO 混乱

- [ ] 部分地方直接使用 Entity 作为响应体，泄露了不该返回的字段
- [ ] 建议严格区分 Entity / DTO / VO 三层

### 缓存序列化

- [ ] Redis 缓存手动序列化/反序列化（JSON），缺少统一的缓存管理器封装
- [ ] 考虑使用 Spring Cache + RedisTemplate 泛型封装，减少样板代码
